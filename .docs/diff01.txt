+# Issue04: listup_json返り値のPath化
+
+## 背景
+
+PR03で`infra.filesys.listup_json()`からPROJECT_ROOT依存を排除したが、
+返り値の型は`list[str]`となっている。
+
+この結果、呼び出し側（presentation / application）が
+
+- 文字列をPathとしてみなして扱う
+- 後続処理で再度`Path`に変換する
+
+といったコードを書かざるを得ず、責務境界と抽象度が曖昧になっている。
+
+PR03内の「発見した問題点」としても、
+`listup_json() -> list[Path]` への変更が提案されている。
+
+## 問題点
+
+1. `listup_json()`の返り値が`list[str]`であり、型が**Pathそのもの**を表現していない。
+2. CLI層がPath相当の情報を文字列で扱うため、抽象度が下がる。
+3. application層・ユースケース層で「文字列 -> Path」の再変換が必要になり、重複ロジック・バグの温床となる。
+
+## 目的
+
+- `listup_json()`の返り値を`list[Path]`に変更し、
+  「ファイルパスを列挙する」という責務をinfra層に集約する。
+- presentation / application層ではPathオブジェクトをそのまま扱い、
+  表示などの都合で文字列化が必要な場合のみ変換を行うようにする。
+- レイヤーごとの抽象度と責務境界を明確にする。
+
+## 方針
+
+- `infra.filesys.listup_json()`の返り値型を`list[str]`から`list[Path]`に変更する。
+- `application.select.json_usecase.resolve_choice()`の引数・返り値の型を
+  `list[str]` / `str` から `list[Path]` / `Path` に変更する。
+- `presentation.cli.select_json.select_json()`では、
+  Pathのリストを受け取り、
+  - ユーザー表示時のみ文字列化
+  - 選択結果としてはPathをそのまま保持
+  するように修正する。
+- 既存テスト（特に `tests/application/select/test_json_usecase.py`）も
+  Path型に合わせて期待値を更新する。
+
+## 完了条件
+
+- `infra.filesys.listup_json()`の返り値が`list[Path]`となっている。
+- `application.select.json_usecase.resolve_choice()`が`list[Path]`を受け取り、`Path`を返す。
+- `presentation.cli.select_json.select_json()`が、
+  - Pathのリストを受け取り
+  - 表示時のみ文字列化している
+  ことが確認できる。
+- `tests/application/select/test_json_usecase.py`がPath型に追従しており、
+  pytestがグリーンで通る。
+
+## スコープ外
+
+- `listup_json()`の列挙ロジック自体（どのディレクトリまで探索するか等）の変更。
+- CLIインターフェースそのものの仕様変更（選択UIの変更など）。

+# PR04: listup_json返り値のPath化
+
+## 問題の概要
+
+PR03で指摘した通り、infra層の`listup_json()`は返り値の型が`list[str]`であり、
+呼び出し側（CLI層やユースケース層）でPathに再変換する必要がある設計だった。
+
+- Pathを扱う責務の境界が曖昧
+- CLI層の抽象度が下がる（文字列でPath相当を扱っている）
+
+この不整合を解消するため、Issue04として`listup_json() -> list[Path]`への変更を行った。
+
+
+## infra/filesys.py
+
+**変更前**
+```python
+from pathlib import Path
+
+
+def listup_json(base_path: Path, dir_name: str) -> list[str]:
+	d_path = base_path / dir_name
+
+	if not d_path.exists() or not d_path.is_dir():
+		raise FileNotFoundError(f"ディレクトリが見つかりません: {dir_name}")
+
+	files = [
+		str(p.relative_to(base_path / dir_name))
+		for p in d_path.rglob("*.json")
+		if p.is_file()
+	]
+
+	if not files:
+		raise FileNotFoundError("JSONファイルが見つかりません")
+
+	return sorted(files)
+```
+
+**変更後**
+```python
+from pathlib import Path
+
+
+def listup_json(base_path: Path, dir_name: str) -> list[Path]:
+	d_path = base_path / dir_name
+
+	if not d_path.exists() or not d_path.is_dir():
+		raise FileNotFoundError(f"ディレクトリが見つかりません: {dir_name}")
+
+	files = [
+		p.relative_to(d_path)
+		for p in d_path.rglob("*.json")
+		if p.is_file()
+	]
+
+	if not files:
+		raise FileNotFoundError("JSONファイルが見つかりません")
+
+	return sorted(files)
+```
+
+**説明**
+
+- 返り値を`list[str]`から`list[Path]`に変更。
+- これにより、「どの層がPathを扱うか」の責務をinfra層に集約できる。
+- 呼び出し側はPathのまま受け取り、必要に応じて文字列化するだけでよい。
+
+
+## application/select/json_usecase.py
+
+**変更前**
+```python
+def resolve_choice(files: list[str], index: int) -> str:
+	if index < 1 or index > len(files):
+		raise ValueError("選択した値が不正です。")
+
+	return files[index - 1]
+```
+
+**変更後**
+```python
+from pathlib import Path
+
+
+def resolve_choice(files: list[Path], index: int) -> Path:
+	if index < 1 or index > len(files):
+		raise ValueError("選択した値が不正です。")
+
+	return files[index - 1]
+```
+
+**説明**
+
+- `listup_json()`の返り値の型変更に追従し、`files`を`list[Path]`として扱うようにした。
+- ユースケース層ではPathそのものを扱い、具体的な文字列表現はさらに外側（CLI層）に委ねる。
+
+
+## presentation/cli/select_json.py
+
+**変更前（概略）**
+```python
+@click.command(name="select-json")
+@click.pass_obj
+def select_json(config: AppConfig) -> None:
+	dir_name = config.json_dir
+	base_path = config.base_path
+	files = listup_json(base_path, dir_name)  # list[str]
+
+	click.echo("\n翻訳対象のファイルを選択せよ:")
+	for i, p in enumerate(files, 1):
+		click.echo(f"{i}: {p}")
+
+	choice = click.prompt("番号", type=int)
+
+	selected = resolve_choice(files, choice)  # str
+
+	click.secho(f"選択されたファイル: {selected}")
+```
+
+**変更後（概略）**
+```python
+@click.command(name="select-json")
+@click.pass_obj
+def select_json(config: AppConfig) -> None:
+	dir_name = config.json_dir
+	base_path = config.base_path
+	files = listup_json(base_path, dir_name)  # list[Path]
+
+	click.echo("\n翻訳対象のファイルを選択せよ:")
+	for i, p in enumerate(files, 1):
+		click.echo(f"{i}: {p}")  # 表示時のみ文字列化
+
+	choice = click.prompt("番号", type=int)
+
+	selected = resolve_choice(files, choice)  # Path
+
+	click.secho(f"選択されたファイル: {selected}")
+```
+
+**説明**
+
+- CLI層ではPathを受け取り、ユーザー向けの表示時にのみ暗黙的に文字列化している。
+- 選択結果としてはPathを保持するため、後続処理（実際のファイル読み込みなど）で再変換が不要になる。
+- これにより、CLI層の抽象度を保ちつつ、infra/アプリケーション層でPathを一貫して扱えるようになった。
+
+
+# 影響範囲
+
+- `infra.filesys.listup_json()`の返り値型変更（`list[str]` -> `list[Path]`）
+- `application.select.json_usecase.resolve_choice()`のシグネチャ変更
+- `presentation.cli.select_json.select_json()`の呼び出し部・型整合性の修正
+- `tests/application/select/test_json_usecase.py` の期待値更新（文字列からPathオブジェクトへ）
+
+いずれもPath型への置き換えに限定されており、
+ロジック自体（選択番号の検証やファイル列挙アルゴリズム）は変更していない。
+
+
+# 設計的な効果
+
+- Pathを扱う責務がinfra/ユースケース層に集約され、CLI層の抽象度が適切に保たれる。
+- 後続処理での「文字列 -> Path」再変換が不要になり、コード量とバグ混入余地が減少。
+- PR03で指摘した「返り値の不整合」を解消し、レイヤー間の責務境界が明確になった。
